**Step 2 — Create `.github/workflows/deploy.yml`:**

```yaml
name: Deploy Hugo site to Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: false

defaults:
  run:
    shell: bash

jobs:
  build:
    runs-on: ubuntu-latest
    env:
      HUGO_VERSION: 0.160.1
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          submodules: recursive
          fetch-depth: 0

      - name: Install Hugo
        uses: peaceiris/actions-hugo@v3
        with:
          hugo-version: ${{ env.HUGO_VERSION }}
          extended: true

      - name: Setup Pages
        id: pages
        uses: actions/configure-pages@v5

      - name: Build with Hugo
        run: hugo --minify --gc

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./public

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

Notes:
- `HUGO_VERSION: 0.160.1` pins to the version verified locally (`hugo version` output: `hugo v0.160.1+extended+withdeploy darwin/arm64`). `extended: true` because rewired-overrides.css builds rely on Hugo Extended's SCSS toolchain (though current code uses plain CSS, Phase 1 set this up via `css.Build` which needs extended).
- `peaceiris/actions-hugo@v3` is the canonical community action for pinning Hugo in CI.
- `submodules: recursive` because the theme is vendored in `themes/arcaeon/` but may be tracked as a submodule in some Hugo setups (Phase 1 confirmation: theme was vendored read-only as a directory, no submodule — `recursive` is a no-op safety flag but harmless).
- Native GH Pages pipeline (`configure-pages` → `upload-pages-artifact` → `deploy-pages`) chosen over a third-party gh-pages branch push — matches D-10's primary option.
- Workflow runs on every `push` to `main` and can be triggered manually (`workflow_dispatch`) — satisfies DEPL-05.
